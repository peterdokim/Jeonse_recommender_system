가설 2: 복합 안전 + ML
단일 지표로는 안전을 잴 수 없다 — 3축 복합 + ML 예측 + 백테스트 삼중 검증
① 3축 복합 안전 점수
S_RATE (50%) ← 전세가율 백분위
S_MIG (25%) ← 거래량 백분위
S_SUB (25%) ← 변동성 백분위
A(≥80) / B(≥60) / C(≥40) / D
▼
RULE_SCORE
0.6×안전 + α/β/γ/δ × 보조점수
α:손실회피 β:가격적정 γ:선호적합 δ:유사도 (합 1.0)
▼
RECOMMENDATION_SCORE
= RULE_SCORE × 0.80 + ML_SAFETY × 0.20
▲
② GradientBoosting ML
8개 피처 · ROC AUC ≈ 0.74
ML_SAFETY = 100 − ML_RISK_SCORE
· 전세가율 70%여도 거래 5건 미만이면 유동성 위험
· 과거 안정성은 규칙, 미래 변동성은 ML, 회복력은 백테스트
· 룰과 ML을 블렌딩하고 백테스트로 추가 필터링
③ 휴리스틱 백테스트
롤링 12개월 윈도우 가격 시계열 분석
BT_WORST_DRAWDOWN_PCT
최악 낙폭 (35%)
BT_RECOVERY_RATE
12개월 후 회복 비율 (25%)
BT_DOWNSIDE_HIT_RATE
−10%+ 하락 빈도 (15%)
BACKTEST_SCORE
0.35×worst + 0.25×median + 0.25×recovery + 0.15×(100−hit)
④ Hard Filter 4중 적용
· 예산: 보증금 + 허용범위% 이하
· 지역 범위: 관심 구 / 출퇴근권 / 전체
· 거래량: 최근 6개월 5건 미만 제외
(HAS_ENOUGH_TX)
· 절대 위험: 전세가율 90% 이상 절대 제외
(NOT_ABSOLUTE_RISK)
ML 피처: JEONSE_RATE, SALE_CUSHION_PCT, RATE_VOLATILITY, JEONSE_CHANGE_12M_PCT, RECENT_TX_COUNT, TRADE_COUNT, RENT_COUNT, TRADE_RATIO_PCT
타겟: LEAD(JEONSE_PRICE,6)로 6개월 후 5%+ 하락 라벨 · n_estimators=200 max_depth=4 lr=0.05 · 80/20 stratified

===== SLIDE 13 =====
느낀점 / 회고
프로젝트를 통해 배운 것
기술적으로 배운 점
Snowflake 하나로 정말 다 됐다.
ETL부터 ML 학습/추론, LLM 호출까지 한 곳에서 끝나는 게 처음이라 신기했다.
AI_AGG가 인상적이었다.
SQL 한 줄 안에서 LLM을 집계 함수처럼 쓰는 건 처음 봤는데, 데이터베이스가 자연어 인사이트를 바로 뱉어내는 게 신선했다.
Snowpark ML Registry는 실용적이었다.
모델을 등록하고 Snowflake 안에서 바로 추론까지 돌릴 수 있어서 데이터를 밖으로 빼낼 필요가 없었다.
문제 정의에서 배운 점
뭘 만들지 정하는 게 제일 어려웠다.
처음엔 '전세가 안전한지 알려주는 앱'을 만들려고 했는데, 사용자가 진짜 궁금한 건 '그래서 다른 어디가 더 나은데?'였다. 중간에 방향을 틀어서 진단이 아니라 추천으로 다시 설계했다.
숫자보다 말이 더 잘 통한다.
'전세가율 78%, 등급 B' 같은 숫자보다 LLM이 한 문장으로 풀어 써 준 설명이 훨씬 와닿았다.
처음 짠 추천 로직은 잘 안 됐다.
